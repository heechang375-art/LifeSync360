■ 현재 스펙이 적절한 부분

1) EC2 t3.small (서울, MySQL/Tokenization/Private API/Ansible)
   프로젝트 및 테스트 수준에서 비용 대비 합리적인 선택이다.
   실서비스 전환 시에는 워크로드별로 아래 방향으로 분리가 필요하다:
   - MySQL:         r7g.large (메모리 집약)
   - Tokenization:  c7g.medium (CPU 집약, SHA-256)
   - Private API:   t3.small 유지 (I/O 바운드, 내부 전용)
   - Ansible Node:  t3.micro로 다운 (간헐적 실행)

2) DynamoDB On-Demand 모드
   배치성 요청 패턴에 맞는 과금 구조다. 유휴 비용 없음.

3) S3 Standard 30GB
   Raw → Processed → Curated 3단계 파이프라인에 적합한 구성이다.

4) ALB + WAF 조합
   외부 트래픽 수신 + SQLi/XSS 차단 + IP 기반 제어가 모두 포함된 적절한 구성이다.

5) Secrets Manager 서울 4개 구성
   DB통합, GCP키, Glue환경변수, KMS ARN — 실제 필요한 시크릿 분류가 맞다.


■ 최적화 가능한 부분

1) EMR 실시간 m4.large → m6g.large 교체 [즉시 권장]
   m4는 2015년 출시 구형 인스턴스다. 동급 스펙에서 32% 비싸다.

   m4.large:  2 vCPU / 8GB RAM / $0.114/hr (x86, 2015년)
   m6g.large: 2 vCPU / 8GB RAM / $0.077/hr (Graviton2, 2020년)
   절감: 월 $2.70 → 약 $1.85 (약 $0.85/월, 연 $10 절감)
   변경 방법: EMR 클러스터 인스턴스 유형 설정에서 교체

2) Aurora db.t3.medium → Serverless v2 [권장]
   현재 45시간/월 사용 = 하루 약 1.5시간만 사용하는 배치성 패턴이다.
   이 패턴에서 프로비저닝 인스턴스는 유휴 시에도 비용이 나간다.

   db.t3.medium 온디맨드: $0.073/hr × 45h = $3.29 (인스턴스만)
   Serverless v2 (0.5 ACU 최소): $0.06/ACU-hr × 0.5 × 45h = $1.35
   절감: 약 $4/월 → 연 $50 절감 + 유휴 시 scale-to-zero 가능 (2024.11 기준)

3) Lambda x86 → arm64 [권장]
   Lambda 아키텍처를 arm64로 변경하면 동일 성능 대비 20% 저렴하다.
   현재 무료 범위 내라 금액 영향은 미미하지만,
   트래픽 증가 시 비용 효율 차이가 누적된다.
   변경 방법: Lambda 함수 설정 > 아키텍처 > arm64 선택 (코드 변경 불필요)

4) ECS Fargate x86 → arm64 [권장]
   taskdef.json에 아래 추가 시 20% 비용 절감:
   "runtimePlatform": {"cpuArchitecture": "ARM64", "operatingSystemFamily": "LINUX"}
   현재 $2.56 → 약 $2.05로 절감

5) ECS Fargate 256CPU/512MB → 512CPU/1024MB [운영 전환 시]
   현재 스펙은 Fargate 최소 구성이다.
   FastAPI 기동 시 기본 100~200MB를 사용하므로
   DB 커넥션 풀 + 동시 요청 증가 시 512MB에서 OOM이 발생할 수 있다.
   운영 전환 시 512CPU/1024MB로 상향하고 Auto Scaling과 조합 권장.


■ 발표 시 예상 질문 대응

Q. "t3.small로 MySQL을 운영하는 게 맞나요?"
A. "현재는 프로젝트 수준으로 비용 최적화를 우선했습니다.
   t3.small은 2 vCPU / 2GB RAM으로, 웨어러블 스트리밍 write와
   PII 임시 보관이 동시에 발생하는 이 구조에서
   실서비스 전환 시에는 메모리 최적화 인스턴스(r7g.large)로
   교체가 필요합니다. 현재 스펙은 테스트 및 개발 단계 기준입니다."

Q. "EMR 인스턴스 타입이 두 종류인 이유가 뭔가요?"
A. "초기 대용량 데이터 적재(역사 데이터 일괄 처리)에는
   m6g.xlarge 4대로 처리 속도를 확보하고,
   이후 일상 운영 시 실시간 처리는 m4.large 2대로 비용을 낮췄습니다.
   다만 m4는 구형 인스턴스라 m6g로 통일하면 32% 비용 절감이 가능합니다."

Q. "Aurora를 Serverless로 하지 않은 이유가 있나요?"
A. "현재는 db.t3.medium 프로비저닝으로 구성했습니다.
   45시간/월 사용 패턴에서는 Aurora Serverless v2가
   비용상 더 유리하며, 2024년 11월부터 scale-to-zero도 지원합니다.
   실서비스 전환 시 Serverless v2로 교체를 권장합니다."

2. ML vs LLM 선택 이유
----------------------------------------------------------------

■ 결론
이 프로젝트의 use case(건강 점수, 금융/보험 추천)는 ML이 LLM보다 명백히 맞는 선택이다.

■ 이유 1 — 데이터 타입 미스매치
LLM은 텍스트/언어 패턴 처리용이다.
이 프로젝트의 입력 데이터는 심박수, SpO2, HRV, 체온(시계열 수치), 나이, 성별,
거래 이력(정형 테이블), 보험 가입 여부, 포인트 잔액(카테고리/숫자)이다.
정형 tabular data에는 XGBoost, LightGBM 같은 ML 모델이 훨씬 정확하고 효율적이다.

■ 이유 2 — 비용 격차
항목          ML                           LLM
추론 비용     수십만 유저 배치 → 수 달러   토큰 기반 → 수백~수천 달러
모델 크기     수 MB                        수 GB ~ 수백 GB
GPU 요구      없거나 최소                   필수

■ 이유 3 — 설명가능성 (Explainability)
건강 점수(0~100) 산출, 보험 할인율 적용에는 근거가 필요하다.
ML: SHAP value로 "심박 변이도가 점수 하락에 가장 영향" 설명 가능
LLM: 왜 그 점수인지 설명 불가 → 금융/헬스케어 규제 환경에서 치명적

■ 이유 4 — 일관성과 재현성
같은 입력에 대해 항상 같은 출력이 보장되어야 한다.
ML 모델은 완전히 결정적(deterministic)이고, LLM은 확률적으로 매번 달라진다.

■ 이유 5 — 개인정보 보호
LLM API에 건강 데이터 + 금융 데이터를 전송하면 컴플라이언스 문제가 발생한다.
Vertex AI Custom Training은 데이터가 외부 LLM 서비스로 나가지 않는다.

■ 한 줄 요약
건강 점수 계산과 상품 추천은 언어 생성 문제가 아니라 분류/회귀/랭킹 문제다.
ML이 도구로서 맞고, LLM은 이 use case에 구조적으로 부적합하다.

3. VPC 다중 분리 — 설계 이유 및 질의응답
----------------------------------------------------------------

■ 현재 VPC 구성 (실제 확인된 구성)

VPC              리전                역할
On-Prem VPC      ap-northeast-2 서울  MySQL, Ansible — PII 원본
Service DB VPC   ap-northeast-2 서울  Aurora, DynamoDB, Lambda, RDS Proxy
Data VPC         ap-northeast-2 서울  S3, Glue, EMR — 정제 파이프라인
Management VPC   ap-northeast-2 서울  SSM, Bastion — 운영 채널
Wearable VPC     ap-northeast-3 오사카 IoT 웨어러블 데이터 수신
Group VM VPC     ap-northeast-3 오사카 LS 계열사 시스템 연동

서울 ↔ 오사카 연결: Site-to-Site VPN 2개 ($8.64/월)


■ Q. VPC를 왜 이렇게 많이 나눴나요? 서브넷으로도 충분하지 않나요?

A. 서브넷 분리는 논리적 분리고, VPC 분리는 물리적 분리입니다.
   서브넷은 같은 라우팅 테이블과 인터넷 게이트웨이를 공유하기 때문에
   보안 그룹 설정 실수 하나로 전체가 뚫릴 수 있습니다.
   VPC는 라우팅 자체가 없으면 물리적으로 통신이 불가능합니다.
   금융/헬스케어 데이터를 다루는 이 서비스는 ISMS-P, 금융보안원 가이드상
   "논리적 분리"로는 감사를 통과하지 못합니다.


■ Q. VPC마다 데이터 민감도가 다르다는 게 무슨 의미인가요?

A. 각 VPC가 보유한 데이터의 민감도와 외부 노출 가능성이 다릅니다.

   On-Prem VPC    → 주민번호, 계좌번호 원본 (PII 최고 등급, 외부 접점 없음)
   Service DB VPC → AES-256 암호화된 PII + 금융 추천 결과 (Lambda만 접근)
   Data VPC       → 비식별 처리된 분석용 데이터 (Glue Job만 접근)
   Wearable VPC   → 외부 IoT 기기에서 직접 데이터 수신 (외부 접점 최대)

   민감도가 낮은 VPC(Wearable)가 공격받아도
   라우팅이 없으면 민감도 높은 VPC(On-Prem, Service DB)로 이동 자체가 불가능합니다.


■ Q. VPC를 나누면 비용이 더 들지 않나요?

A. TGW 비용이 발생합니다. 서울 내 TGW 1개: $12.70/월 (180시간 기준)
   이 비용은 보안 사고 발생 시 대응 비용, 규제 위반 과징금,
   서비스 중단 손실과 비교하면 보험료 수준입니다.
   실제 금융/헬스케어 서비스에서 VPC 분리는 비용이 아니라 필수 요건입니다.


■ Q. TGW 대신 VPC Peering을 쓰면 더 싸지 않나요?

A. VPC가 3~4개 이하면 Peering이 낫습니다.
   6개 이상이면 Peering은 N*(N-1)/2 개 연결이 필요해 15개 연결을 각각 관리해야 합니다.
   TGW는 허브 앤 스포크 구조로 중앙에서 라우팅을 통제하므로
   VPC 수가 많을수록 TGW가 관리 면에서 유리합니다.


■ Q. Wearable VPC와 Group VM VPC가 오사카 리전에 있는 이유가 뭔가요?

A. 외부 접점 시스템을 중앙 서울 리전 인프라와 물리적으로 분리하기 위한 설계였습니다.
   다만 이 설계에는 해결해야 할 이슈가 있습니다.

   [문제점]
   AWS에서 서울(ap-northeast-2) 외 다른 리전은 전부 국외입니다.
   오사카(ap-northeast-3)는 일본 리전입니다.
   웨어러블에서 수집되는 심박수, SpO2, 체온은 민감정보(건강정보)로,
   국외 리전을 경유하면 개인정보보호법상 국외 이전 동의 또는 인가가 필요합니다.

   또한 AWS 공식 아키텍처 패턴에서 "외부 시스템 시뮬레이션을 위해
   다른 리전 VPC를 사용하는" 케이스는 존재하지 않습니다.
   멀티 리전 VPC는 실제 글로벌 서비스의 Active-Active/DR 구성에 사용됩니다.

   [올바른 방향]
   - 웨어러블: AWS IoT Core(서울, 퍼블릭 엔드포인트) → 서울 Wearable VPC(프라이빗)
   - 계열사:   Site-to-Site VPN → 서울 TGW → Group VM VPC(서울)
   외부 경계는 다른 리전이 아닌 IoT Core + VPN으로 표현하는 것이 맞습니다.

   [현재 구성에서의 실용적 대응]
   오사카 → 서울 VPN 연결 구간에서 건강 데이터를 저장하지 않고
   수신 즉시 서울로 스트리밍하여 오사카에 원본이 남지 않도록 처리하면
   국외 이전 리스크를 최소화할 수 있습니다.

4. GCP 리소스 사용 이유 (멀티 클라우드 전략)
----------------------------------------------------------------

■ 실제 데이터 흐름 (Excel + 아키텍처 기준)

AWS On-Prem VPC → S3 Raw → Glue/EMR (정제) → S3 Curated
                                                      ↓
                                          Storage Transfer Service
                                                      ↓
                        GCS → Dataflow → BigQuery → Vertex AI
                                                      ↓
                              Eventarc → Cloud Run → AWS API GW
                                                      ↓
                                         Lambda → Aurora / DynamoDB

정제는 AWS에서 하고, 정제된 비식별 데이터만 GCP로 전송한다.
크로스 클라우드 전송(S3 Curated → GCS)은 어느 선택을 해도 발생한다.


■ Redshift/SageMaker도 충분히 강력하다 (솔직한 전제)

Redshift Serverless 2025 기준:
- 최대 1024 RPU, 최대 8PB, 동시 쿼리 50개 대기열
- 쿼리당 최소 60초 과금

SageMaker:
- Multi-model Endpoint 지원 (Vertex AI는 기본 미지원)
- 더 성숙한 플랫폼, 풍부한 레퍼런스

"AWS만으로 기술적으로 불가능한 것은 없다."
GCP를 선택한 이유는 성능이 아니라 파이프라인 구조와 규정이다.


■ BigQuery를 선택한 실제 이유

1) GCS 도착 이후 추가 복사 없이 Vertex AI까지 완결
   BigQuery + Vertex AI: GCS → BigQuery → Vertex AI (추가 복사 없음)
   Redshift + SageMaker: Redshift(분석) + S3 유지(SageMaker 학습) — 이중 관리

2) 동시 쿼리 처리
   Redshift: 50개 대기열 제한 → 배치 분석 잡 병목 가능
   BigQuery: 슬롯 자동 배분, 실질적 제한 없음

3) 컬럼 수
   Redshift: 테이블당 최대 1,600개
   BigQuery: 테이블당 최대 10,000개

4) 과금 구조
   Redshift Serverless: 쿼리당 최소 60초 과금 → 단기 쿼리에 불리
   BigQuery: 처리 데이터 용량($5/TB) 과금 → 이 프로젝트 패턴에 유리


■ 멀티 클라우드를 선택한 이유

1) 규정 준수와 데이터 주권 [가장 중요]
   원본 PII, 금융 데이터 → AWS 서울 리전 보관 (개인정보보호법 준수)
   비식별 분석 데이터만 GCP 이전 → 국외 이전 규정 리스크 없음

2) 기존 인프라 활용
   계열사들이 이미 AWS 기반 시스템을 운영하고 있어
   전면 GCP 마이그레이션은 비용과 위험이 너무 크다.

3) 워크로드별 과금 모델 최적화
   상시 운영 트랜잭션 DB/API → AWS (예약 인스턴스 활용)
   간헐적 ML 학습/대용량 분석 → GCP 온디맨드 (유휴 비용 없음)

4) 업계 현황
   2025년 기준 87~89% 엔터프라이즈가 2개 이상 클라우드 사용.
   멀티 클라우드는 특수한 선택이 아닌 업계 표준이다.

■ 핵심 요약
"GCP가 AWS보다 낫기 때문"이 아니라
"규정상 AWS를 유지해야 하고, 분석/ML 파이프라인은 GCP 생태계에서
 데이터 이동 없이 완결되어 운영이 단순하기 때문"이다.


----------------------------------------------------------------
5. Vertex AI vs Gemini — "Gemini로 바꾸면 안 되나?" 질문 대응
----------------------------------------------------------------

■ 현재 상황 (2026년 4월 기준)
Google Cloud Next 2026에서 Vertex AI가 "Gemini Enterprise Agent Platform"으로
리브랜딩됐다. Vertex AI 기능은 그대로 유지되며 코드 변경 없이 동일 동작한다.
"Vertex AI가 없어지고 Gemini로 대체된다"는 오해다.

■ 두 도구가 푸는 문제 자체가 다르다

Vertex AI AutoML
  목적:  데이터를 학습해서 숫자/분류값을 예측하는 모델
  입력:  BigQuery 테이블, CSV (정형 데이터)
  출력:  점수(숫자), 등급(분류), 순위(랭킹)
  특징:  결정적(동일 입력 = 동일 출력), 설명가능, 저비용 배치

Gemini API
  목적:  프롬프트에 대한 자연어 텍스트를 생성하는 것
  입력:  텍스트, 이미지 (비정형)
  출력:  생성된 텍스트
  특징:  확률적(매번 다름), 설명 불가, 토큰 과금

■ 건강 점수에 Gemini를 쓰면 안 되는 이유

Vertex AI AutoML:
  입력 {heart_rate: 72, spo2: 97, hrv_ms: 45} → 출력 {score: 78, grade: "GOLD"}
  - 10만 명 데이터 학습 → 패턴 내재화 → 숫자 예측
  - 항상 같은 결과, 이유 설명 가능

Gemini API:
  "심박수 72, SpO2 97인 사람의 건강 점수는?" → "약 75~80점으로 추정됩니다"
  - 언어 패턴으로 그럴듯한 숫자를 생성 (패턴 학습 아님)
  - 매번 다른 점수, 근거 없음, 100만 건 처리 시 수천 달러

■ Gemini가 의미 있는 경우 (딱 하나)
  추천 이유 자연어 생성: "최근 HRV 수치가 15% 하락하여 심혈관 주의가 필요합니다"
  → 하지만 규칙 기반 템플릿으로 충분히 대체 가능 (Gemini API 비용 불필요)

■ 발표 시 예상 질문 대응

Q. "Vertex AI가 오래됐는데 최신 Gemini를 쓰는 게 낫지 않나요?"
A. "Vertex AI는 2026년 4월 Gemini Enterprise Agent Platform으로 통합됐고
   기능은 그대로 유지됩니다. Gemini API는 텍스트 생성 도구이고
   Vertex AI AutoML은 정형 데이터에서 숫자를 예측하는 도구입니다.
   건강 점수 계산은 언어 생성 문제가 아니라 수치 예측 문제이기 때문에
   Vertex AI AutoML이 맞는 선택입니다."

Q. "Gemini도 숫자 출력이 되지 않나요?"
A. "출력 형식은 가능하지만 근거가 없습니다.
   금융/헬스케어 서비스에서는 같은 입력에 같은 결과가 나와야 하고,
   왜 그 점수인지 설명할 수 있어야 규제 요건을 충족합니다.
   Gemini는 이 두 가지를 보장하지 않습니다."

■ 한 줄 요약
Vertex AI = 데이터로 패턴을 학습해서 숫자를 예측
Gemini    = 언어 패턴으로 텍스트를 생성
이 프로젝트는 전자가 필요하다.
