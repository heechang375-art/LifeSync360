## 작업 정리 요청 처리

"진행할거 정리해", "순서 알려줘", "어떻게 하면 돼" 같은 요청이 오면:

- **맥락을 먼저 파악해.** 대화 흐름에서 어떤 범위(온프레미스인지, 클라우드인지, 특정 기능인지)를 묻는지 확인해. 대화 맥락이 온프레미스면 온프레미스만, 특정 작업이면 그것만 정리해.
- **완료된 것은 넣지 마.** 앞으로 할 것만 정리해. 히스토리나 완료 항목을 섞으면 안 돼.
- **가이드 형식으로 써.** 파일만 보고 그대로 실행할 수 있도록 순서, 명령어, 확인 방법까지 포함해. "1번 하고 2번 해" 수준의 목록이 아니라 각 단계에 실제 명령이 있어야 해.
- **내가 직접 할 수 없는 것(AWS 콘솔, VM 직접 접속 등)은 명확히 구분해.** "네가 실행" vs "내가 작성할 파일" 섞지 마.

## 코딩 전에 먼저 생각하기

**가정하지 마. 혼란을 숨기지 마. 트레이드오프를 드러내.**

구현 전에:
- 가정한 것들을 명시적으로 밝혀. 불확실하면 먼저 물어봐.
- 해석이 여러 가지라면 조용히 하나 선택하지 말고 선택지를 제시해.
- 더 단순한 방법이 있으면 말해. 필요하면 반박도 해.
- 뭔가 불명확하면 멈춰. 뭐가 헷갈리는지 짚고 물어봐.

**아키텍처 의도가 불명확하면 무조건 먼저 물어봐.**

다음 상황에서는 코드 작성 전에 반드시 확인해:
- 데이터가 어느 저장소에 들어가야 하는지 (DB vs 캐시 vs 파일 등)
- 시스템 간 데이터 흐름이 요청에 명시되지 않은 경우
- 여러 컴포넌트가 같은 데이터를 다루는 경우 (역할 분리 의도 확인)
- 기존 아키텍처 결정과 충돌할 수 있는 구현을 하려는 경우
- 기존 도구/인프라의 동작 방식을 직접 확인하지 않고 가정하는 경우 — "이렇게 동작하겠지"라는 추측으로 코드 짜지 마. 모르면 물어봐.

구현 후에 방향이 바뀌면 두 배로 일하게 돼. 30초 질문이 30분 재작업을 막아.

**코드 수정 전에 변경 내용을 설명하고, 반드시 확인을 받은 후에 시작해.**

UI/기능 변경 시 파일을 열기 전에 다음을 텍스트로 먼저 말해:
- 어떤 파일을 바꾸는지
- 각 파일에서 무엇을 어떻게 바꾸는지
- 왜 그렇게 해석했는지

설명 후 반드시 멈추고 사용자의 "응", "ㅇㅇ", "진행해" 같은 명시적 승인을 기다려.
설명만 하고 바로 Edit/Write 도구를 쓰는 건 확인을 받지 않은 것과 같아 — 절대 하지 마.
승인 없이 진행하면 방향이 틀렸을 때 크레딧만 낭비하고 재작업하게 돼.

## 코드 작성 원칙

**최소한의 코드로 문제를 해결해. 추측성 코드는 넣지 마.**

- 코드는 사람이 직접 작성한 것처럼, 실용적이고 군더더기 없게
- 불필요한 주석, 빈 줄, 보일러플레이트 코드 절대 넣지 마
- 요청하지 않은 기능, 추상화, 유연성, 설정 옵션 추가하지 마
- 불가능한 시나리오에 대한 에러 핸들링 넣지 마
- 200줄로 작성했는데 50줄로 될 것 같으면 다시 써
- 변수명과 함수명은 의도가 명확하게 드러나도록
- 복잡한 로직은 작은 단위로 쪼개고, 재사용 가능하게 설계
- 코드 작성 전에 전체 구조와 접근 방식 먼저 설명해
- 라이브러리/툴/플랫폼 사용 시 공식문서 기준으로 작성해. 기억에 의존하지 마 — deprecated API, 바뀐 파라미터 쓰는 거 방지
- 공식문서 참고 시 `docs-refs.md`에 툴명/버전/URL/확인한 핵심 내용 기록해. 같은 프로젝트 내에서 다시 참고할 일 있으면 공식문서 가기 전에 여기 먼저 확인해

## 최소한의 변경 (Surgical Changes)

**꼭 필요한 것만 건드려. 네가 만든 것만 정리해.**

기존 코드를 수정할 때:
- 요청과 관계없는 코드, 주석, 포맷팅 "개선"하지 마
- 안 망가진 건 리팩토링하지 마
- 기존 코드 스타일이 맘에 안 들어도 맞춰
- 관련 없는 죽은 코드가 보이면 언급은 해도 삭제는 하지 마

네 변경으로 인해 생긴 불필요한 것들은 정리해:
- 네 수정으로 인해 사용되지 않게 된 import/변수/함수는 제거해
- 원래부터 있던 죽은 코드는 요청이 없으면 건드리지 마

기준: 변경한 모든 줄이 사용자의 요청으로 직접 추적 가능해야 해.

**성공 기준을 정의하고, 검증될 때까지 루프를 돌아.**

여러 단계의 작업은 계획을 먼저 말해:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

## Git
- 커밋 메시지는 명확하게, 변경 이유가 드러나게
- 하나의 커밋에 여러 목적 섞지 마
- Git에 올리기 전 아래 항목이 포함됐는지 반드시 확인해:
  - API 키, 시크릿 키, 액세스 토큰
  - 비밀번호, 인증 정보
  - 개인정보 (이름, 이메일, 전화번호 등)
  - .env 파일 또는 민감 설정 파일
- 위 항목이 코드에 하드코딩돼 있으면 커밋 전에 반드시 경고해줘
- 민감 정보는 환경변수나 Secret Manager로 분리하도록 유도해
- .gitignore에 .env, *.key, *.pem 등 민감 파일이 등록됐는지 확인해줘

## 한국어 프로젝트 주의사항

### 인코딩
- 소스 파일은 항상 **UTF-8 (BOM 없이)** 으로 저장해
- 윈도우 환경에서 작업할 경우 CRLF/LF 혼용 주의 — `.gitattributes`로 통일해
- 로그, 설정 파일, SQL 쿼리에 한글 포함 시 인코딩 명시 (`charset=utf-8`, `CHARSET utf8mb4` 등)

### 문자열 처리
- 한글은 가변 바이트(UTF-8 기준 3바이트) — 길이 계산 시 `len()` 대신 언어별 유니코드 함수 써
- 정규식에서 한글 범위: `[가-힣]` — 완성형 한글만 매칭함, 자모 분리형(`ㄱ`, `ㅏ` 등) 별도 처리 필요
- 문자열 슬라이싱 시 바이트 기준인지 문자 기준인지 항상 확인해

### 정렬 & 검색
- 한글 사전순 정렬은 로케일 설정에 따라 달라져 — `locale` 명시하거나 유니코드 코드포인트 기준 정렬임을 인지해
- DB 검색 시 LIKE 패턴에서 한글 초성 검색은 별도 구현 필요 (표준 SQL로 안 됨)
- Elasticsearch 등 검색엔진 사용 시 한국어 형태소 분석기 (nori, arirang 등) 설정 확인해

### 외부 연동 & API
- 외부 API 호출 시 한글 파라미터는 URL 인코딩 확인해 (`encodeURIComponent` 또는 `urllib.parse.quote`)
- 이메일, SMS, 알림 서비스에 한글 포함 시 EUC-KR/UTF-8 지원 여부 확인해
- 서드파티 라이브러리가 유니코드를 제대로 처리하는지 한글 케이스로 테스트해봐

### 테스트
- 테스트 케이스에 한글 입력값 반드시 포함해 (영문만으로 테스트하면 인코딩 버그 못 잡음)
- 경계값: 빈 문자열, 자모 분리형(`ㅎㅏㄴ`), 한영 혼용, 특수문자 혼합
- CI 환경의 로케일 설정 확인해 — 로컬에선 되는데 CI에서 깨지는 경우 대부분 여기서 남

## 에러 메시지 & 로깅

- 에러 메시지는 한국어로 작성해. 단, 스택트레이스·기술 식별자(errno, exception class 등)는 영어 그대로 둬
- 에러 메시지에는 **무엇이 실패했는지 + 왜 실패했는지** 둘 다 담아. "실패했습니다"만 있는 메시지는 쓰지 마
- Ansible 태스크에 `no_log: true` 쓸 때는 민감 정보가 실제로 노출되는 태스크에만 한정해. 습관적으로 붙이면 에러 파악이 불가능해져
  - `no_log` 붙인 태스크 바로 다음에 디버그용 register + 조건부 출력 태스크 붙이는 패턴 고려해
  - 실패 시 최소한 어떤 태스크에서 어떤 모듈이 실패했는지는 알 수 있도록 `failed_when` / `rescue` 블록으로 커스텀 에러 메시지 남겨
- 로그 레벨 구분해서 써: 운영 중 항상 보여야 하는 건 INFO, 디버깅용은 DEBUG로 분리해 (운영 환경에서 DEBUG 전부 켜두는 거 지양)
