[1] On-Prem 시뮬 — 단일 계정 VPC 격리 방식 정당성
================================================================

■ 환경 제약
단일 AWS 계정만 사용 가능. AWS Organizations 기반 멀티 계정 구성 불가.

■ 핵심 논리
계정 분리 = 청구·IAM 경계 / VPC 분리 = 네트워크 경계
On-Prem 시뮬의 핵심은 "네트워크 격리"이므로 VPC로 충분하다.
VPC는 명시적 라우팅 없이 물리적으로 통신 불가 → 계정이 달라도 동일한 원리.

■ AWS 공식 사례
- AWS TGW Workshop: 동일 계정 내 "Simulated On-Premises VPC"가 표준 실습 구조
  https://catalog.workshops.aws/transit-gateway-with-central-egress
- AWS 블로그: strongSwan으로 동일 계정 EC2를 Customer Gateway로 사용하는 VPN 시뮬
  https://aws.amazon.com/blogs/networking-and-content-delivery/simulating-site-to-site-vpn-customer-gateways-strongswan/
- AWS Well-Architected Labs: 동일 계정 VPC를 외부/On-Prem으로 취급하는 실습 다수
  https://wellarchitectedlabs.com/reliability/

■ On-Prem 특성 구현 현황
  특성              실제 On-Prem          이 설계
  인터넷 직접 노출  없음                  없음 (IGW, NAT GW 없음)
  외부 접근 경로    VPN/Direct Connect    Site-to-Site VPN (시뮬)
  관리 채널         Bastion/점프서버      SSM Session Manager
  배포 방식         Ansible/Jenkins       Ansible (CodePipeline 트리거)
  네트워크 경계     물리적 라우터         VPC (명시적 라우팅만 허용)

■ "CodePipeline으로 On-Prem 자동화하는 게 모순 아닌가?" 반박
AWS CodeDeploy는 On-Premises 배포를 공식 지원한다. (에이전트 설치 방식)
  https://docs.aws.amazon.com/codedeploy/latest/userguide/instances-on-premises.html
AWS Systems Manager Hybrid Activations도 실제 물리 서버를 SSM으로 관리하는 공식 기능이다.
  https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-managedinstances.html
→ "클라우드 CI/CD + On-Prem 배포 대상"은 하이브리드 자동화의 표준 사용 사례.

■ Q&A
Q. "같은 계정 내 VPC면 진짜 On-Prem이라고 볼 수 없지 않나요?"
A. "네트워크 격리 관점에서 VPC는 계정과 무관하게 동일한 격리를 제공합니다.
   AWS TGW Workshop과 공식 블로그에서도 동일 패턴을 표준으로 사용합니다.
   단일 계정 제약이 있기도 하지만, 이 패턴 자체가 AWS 공식 POC 구조입니다."

Q. "CodePipeline으로 On-Prem 환경을 자동화하는 게 앞뒤가 안 맞지 않나요?"
A. "CodeDeploy가 On-Premises 배포를 공식 지원하고,
   SSM Hybrid Activations도 실제 물리 서버 관리를 위한 AWS 공식 기능입니다.
   저희 구조는 하이브리드 배포 자동화의 표준 패턴을 시뮬레이션한 것입니다."

[2] VPC 설계 — 다중 VPC 분리 정당성
================================================================

■ 리뷰어 지적: "VPC 안에 Private Subnet 하나를 DB용으로 쓰는 게 표준 아닌가?"

■ 표준 패턴 vs 우리 설계 비교
  분리 방식       서브넷 분리 (표준)    VPC 분리 (우리 설계)
  네트워크 격리   논리적               물리적
  보안 그룹 실수  전파 가능            전파 불가 (라우팅 없음)
  비용            추가 없음            TGW $12.70/월
  규제 감사 적합  케이스마다 다름      금융/헬스케어 강화 감사 유리

■ VPC 분리를 선택한 이유
ISMS-P, 금융보안원 가이드는 "물리적 분리"를 요구한다.
서브넷 분리는 논리적 격리로, 보안 그룹 설정 실수 하나로 전체가 영향받을 수 있다.
VPC 분리는 라우팅 자체가 없으면 물리적으로 통신 불가.

■ VPC별 데이터 민감도 계층
  On-Prem VPC    → 주민번호·계좌번호 원본 (PII 최고 등급, 외부 접점 없음)
  Service DB VPC → AES-256 암호화된 PII + 금융 추천 결과
  Data VPC       → 비식별 분석 데이터
  Wearable VPC   → 외부 IoT 기기 수신 (외부 접점 최대)
  민감도 낮은 VPC가 침해되어도 라우팅이 없으면 PII VPC로 이동 불가.

■ Q&A
Q. "서브넷으로도 충분하지 않나요?"
A. "금융/헬스케어 규제상 물리적 분리 요건을 충족하려면 VPC 분리가 필요합니다.
   서브넷 분리는 동일 라우팅 테이블을 공유해 설정 실수 시 전파 리스크가 있습니다.
   TGW 비용($12.70/월)은 보안 사고 대응 비용과 비교하면 보험료 수준입니다."

Q. "TGW 대신 VPC Peering이 더 싸지 않나요?"
A. "VPC 3개 이하면 Peering이 유리합니다. 6개 이상이면 N*(N-1)/2 = 15개 연결 필요.
   TGW는 허브 앤 스포크로 중앙 관리, VPC 수가 많을수록 TGW가 관리 면에서 유리합니다."

[3] 오사카 리전 — 외부 경계 분리 목적과 정당성
================================================================

■ 설계 의도
단일 AWS 계정 환경에서 "외부 시스템"과 "메인 인프라"를 가장 강하게 분리하는 방법.
크로스 리전 VPC는 물리적으로 다른 네트워크 인프라 → 명시적 피어링 없이는 통신 불가.
동일 리전 VPC 분리보다 강한 격리이며, Site-to-Site VPN이 "외부망 진입 경로"를 명확히 표현.

■ AWS 공식 사례
- AWS Multi-Region TGW Workshop: 크로스 리전으로 외부 접속 시뮬레이션하는 패턴 포함
  https://catalog.workshops.aws/networking/en-US/advanced/transit-gateway-inter-region-peering
- AWS 블로그: 크로스 리전 TGW Peering으로 외부 연결 표현하는 엔터프라이즈 패턴
  https://aws.amazon.com/blogs/networking-and-content-delivery/building-a-global-network-using-aws-transit-gateway-inter-region-peering/
- AWS IoT Greengrass: 엣지 리전에서 데이터 수집 후 중앙 리전으로 전송하는 공식 패턴
  https://docs.aws.amazon.com/greengrass/v2/developerguide/what-is-iot-greengrass.html

■ LS 그룹 관점 설명
Group VM VPC = LS 그룹 해외 계열사(일본 등)가 한국 메인 인프라에 연결되는 구조 시뮬.
오사카 리전 → 서울 TGW의 VPN 연결 = 국제 전용선 연결의 시뮬레이션.

■ "개인정보 국외 이전 아닌가?" 반박
핵심: 데이터 저장(Storage) ≠ 경유(Transit)

오사카 VPC 처리 방식:
  웨어러블 데이터 수신 → 메모리/스트림 처리 → 즉시 서울로 포워딩
  오사카에 S3, RDS 등 스토리지 서비스 없음 → 원본 데이터 저장 없음.

법적 근거:
  개인정보보호법 제28조의8 "국외 이전"은 국외 서버에 저장하거나 제3자에게 제공하는 행위.
  암호화 경유(transit)만 하고 저장하지 않는 경우는 이전으로 보기 어렵다.
  참고: 개인정보보호위원회 「개인정보의 국외 이전 가이드라인」(2023)
  https://www.pipc.go.kr/np/cop/bbs/selectBoardArticle.do?bbsId=BS074&mCode=C020010000&nttId=8905

추가 보호 장치:
  오사카 ↔ 서울 구간: Site-to-Site VPN (TLS 1.2+ 암호화 터널)

■ Q&A
Q. "Region을 달리 할 이유가 없어 보이는데요?"
A. "단일 계정 환경에서 크로스 리전은 물리적으로 완전히 다른 네트워크 인프라입니다.
   동일 리전 VPC 분리보다 강한 격리를 제공하며,
   Site-to-Site VPN이 '외부망에서 내부망으로 진입하는 경로'를 구조상 명확하게 보여줍니다.
   AWS Multi-Region Workshop에서도 동일한 패턴을 외부 접속 시뮬레이션에 사용합니다."

Q. "건강 데이터가 일본 리전 경유하면 국외 이전 아닌가요?"
A. "오사카 VPC에는 스토리지가 없고 수신 즉시 VPN 암호화 터널로 서울에 포워딩합니다.
   개인정보보호위원회 가이드라인에서 국외 이전은 국외 서버에 저장하거나
   제3자에게 제공하는 경우로, 암호화 경유는 해당하지 않습니다."

Q. "레이턴시가 늘어나지 않나요?"
A. "서울 ↔ 오사카 30~40ms 추가 지연은 맞습니다.
   그러나 Wearable 데이터는 배치성 수집 구조라 UX에 직접 영향을 주지 않습니다."

[4] 멀티 클라우드 전략
================================================================

■ 전제: AWS만으로도 기술적으로 구현 가능하다. GCP를 쓰는 이유 3가지:

① 규정 준수 (가장 중요)
   원본 PII·금융 데이터 → AWS 서울 리전 보관 (개인정보보호법 준수)
   GCP로 이전하는 건 Glue/EMR로 비식별 처리된 분석용 데이터만.
   이름·주민번호·계좌번호는 AWS를 벗어나지 않음.

② 기존 인프라 활용
   계열사(은행·카드·보험)가 이미 AWS 기반 운영 중.
   Cognito 인증, ls_user_id 체계를 그대로 유지하면서 분석/ML만 GCP로 확장.
   전면 GCP 전환보다 현실적이고 비용도 낮음.

③ 파이프라인 단순성
   BigQuery + Vertex AI: GCS 도착 후 추가 복사 없이 GCP 내부에서 ML 완결
   Redshift + SageMaker: 분석용(Redshift) + 학습용(S3) 이중 관리 필요

■ BigQuery vs Redshift 핵심 차이
  항목          BigQuery              Redshift Serverless
  과금          처리 데이터 $5/TB     쿼리당 최소 60초 과금
  동시 쿼리     슬롯 자동 배분        50개 대기열 제한
  컬럼 수       10,000개              1,600개
  ML 연동       SQL로 직접 학습       S3 별도 export 필요
  출처: https://cloud.google.com/bigquery/docs/introduction
  출처: https://docs.aws.amazon.com/redshift/latest/mgmt/serverless-overview.html

■ Vertex AI vs SageMaker 핵심 차이
  Vertex AI 강점: BigQuery 직접 연동, Explainable AI 기본 내장, Eventarc 파이프라인 완결
  SageMaker 강점: Multi-model Endpoint 지원, 성숙한 플랫폼, AWS 통합 유리
  결론: "Vertex AI가 전반적으로 낫다"가 아니라 "BigQuery에 데이터가 있는 이 구조에서
        Vertex AI가 파이프라인 상 더 자연스럽다"
  출처: https://cloud.google.com/vertex-ai/docs/start/introduction-unified-platform

■ 멀티 클라우드 단점 (솔직하게)
  운영 복잡도 상승 / 네트워크 비용 추가 / IAM 이원화 / 장애 추적 어려움
  → 단점은 실재하나, GCP 없이 AWS 단일로 가면 데이터 이중 관리 + 파이프라인 복잡도가 더 높음.

■ Q&A
Q. "멀티 클라우드를 쓰는 이유가 뭔가요?"
A. "GCP가 낫기 때문이 아닙니다. 세 가지입니다.
   ① 원본 PII는 법규상 AWS 서울에 있어야 합니다.
   ② 계열사가 이미 AWS 기반이라 전면 GCP 전환이 불가능합니다.
   ③ 비식별 분석 데이터는 GCP 내부에서 데이터 이동 없이 ML까지 완결됩니다."

Q. "고객 데이터가 AWS → GCP로 이동하는 게 불편하지 않나요?"
A. "GCP로 이전되는 건 비식별 처리된 분석 데이터입니다.
   이름·주민번호·계좌번호는 AWS 서울을 벗어나지 않습니다.
   심박수도 토큰화·일반화 처리 후 수치만 전송합니다."

Q. "GCP를 할 줄 아는 사람이 없으면?"
A. "현실적으로 중요한 지적입니다. 다만 저희가 쓰는 GCP 범위는
   BigQuery + Vertex AI + Cloud Run으로 한정되어 있고,
   모두 managed service라 AWS 경험자가 추가 학습으로 운영 가능한 수준입니다."

Q. "AWS만 써도 되지 않나요?"
A. "기술적으로는 가능합니다. 다만 파이프라인 단순성과
   간헐적 ML 학습 워크로드의 온디맨드 과금 이점을 위해 GCP를 선택했습니다.
   단일 클라우드 운영이 더 중요하다면 Redshift + SageMaker도 유효한 선택입니다."

[5] ML vs LLM 선택 이유
================================================================

건강 점수·보험 추천은 분류/회귀/랭킹 문제 → ML이 맞다.

① 데이터 타입: 심박수, SpO2, HRV, 거래이력 = 정형 tabular data → LLM 부적합
② 비용: ML 배치 수백만 건 = 수 달러 vs LLM 토큰 과금 = 수백~수천 달러
③ 설명가능성: SHAP value로 근거 제시 가능 → 금융/헬스케어 규제 필수
   출처: https://shap.readthedocs.io/en/latest/
④ 재현성: ML = 결정적(같은 입력→같은 출력) vs LLM = 확률적(매번 달라짐)
⑤ 개인정보: Vertex AI Custom Training = 데이터 외부 LLM 서비스로 전송 없음

[6] Vertex AI vs Gemini
================================================================

■ 현황: Vertex AI는 2026.4 "Gemini Enterprise Agent Platform"으로 리브랜딩.
  기능 그대로 유지. "Vertex AI가 Gemini로 대체됐다"는 오해.

■ 역할 차이
  Vertex AI AutoML: 정형 데이터 → 숫자 예측 (결정적, 설명가능, 저비용)
  Gemini API: 자연어 텍스트 생성 (확률적, 설명 불가, 토큰 과금)

■ Q&A
Q. "최신 Gemini를 쓰는 게 낫지 않나요?"
A. "Vertex AI AutoML은 그대로 유지됩니다. Gemini는 텍스트 생성 도구이고,
   건강 점수 계산은 수치 예측 문제라 Vertex AI AutoML이 맞습니다."

Q. "Gemini도 숫자 출력이 되지 않나요?"
A. "출력은 가능하지만 근거가 없습니다. 금융/헬스케어는 같은 입력에 같은 결과와
   설명 가능성이 규제 요건인데, Gemini는 이 두 가지를 보장하지 않습니다."

Q. "t3.small로 MySQL 운영이 맞나요?"

Q. "이 프로젝트를 왜 만드나요?"
A. "LS 계열사(은행·카드·보험)의 고객 데이터가 분절되어 있어
   통합 건강/금융 서비스가 불가능했습니다.
   웨어러블 건강 데이터 + 금융 이력을 결합해 건강 점수(0~100)와
   맞춤 상품을 추천하는 플랫폼입니다."

Q. "비식별화 프로세스가 구체적으로 어떻게 이루어지나요?"
A. "3단계입니다.
   1) On-Prem VPC: SHA-256 토큰화로 PII 원본 → 토큰 치환
   2) Glue/EMR: 웨어러블 수치 일반화(72bpm → 70~75 구간), 집계, 마스킹
   3) S3 Curated: 비식별 완료 데이터만 GCS로 전송
   원본 PII는 On-Prem VPC를 벗어나지 않습니다."

Q. "데이터 유출 시 대응 계획은?"
A. "① 감지: GuardDuty + CloudWatch Anomaly Detection
   ② 격리: SG 즉시 차단 (SSM Automation)
   ③ 통보: 72시간 내 개인정보보호위원회 신고 (법 제34조)
   ④ 복구: Aurora PITR, S3 버전 관리"

Q. "단일 장애점(SPOF)은 어디인가요?"
A. "TGW(AWS 관리형 99.9% SLA), On-Prem MySQL 단일 인스턴스(실서비스 전환 시 Multi-AZ 필요),
   S3→GCS Storage Transfer(재시도 정책과 알람 필수)입니다."

Q. "Glue와 EMR을 둘 다 쓰는 이유는?"
A. "Glue는 서버리스 ETL로 일상 정제 작업에, EMR은 초기 대용량 역사 데이터 적재에 사용합니다."

Q. "AWS와 GCP 모니터링을 어떻게 통합하나요?"
A. "현재는 CloudWatch(AWS)와 Cloud Monitoring(GCP)을 각각 운영하고
   알람만 SNS와 Cloud Alerting에서 Slack으로 통합합니다.
   필요시 Grafana로 양쪽 데이터 소스를 단일 대시보드로 통합 가능합니다."

Q. "ISMS-P 인증을 받을 수 있는 구조인가요?"
A. "핵심 요건 세 가지 중 ①물리적 분리(VPC) ②접근제어·암호화(KMS·SM·AES-256)를 충족합니다.
   ③개인정보 처리 방침 문서화는 별도 준비가 필요합니다."

Q. "배포 전략은?"
A. "ECS Fargate: CodeDeploy Blue/Green / Lambda: Alias Traffic Shifting(Canary)
   On-Prem(EC2): Ansible rolling update (한 대씩 순차, 실패 시 중단)"
