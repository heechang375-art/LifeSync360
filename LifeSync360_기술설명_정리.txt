0. 컴퓨팅 리소스 최적화 — 인스턴스 선택 기준
----------------------------------------------------------------

■ 선택 원칙

1) Graviton3(ARM) 우선 — 동일 워크로드 대비 x86 대비 20% 저렴, 60% 전력 효율
2) 워크로드 특성에 맞는 패밀리 선택
   r계열: 메모리 집약 (DB, 캐시)
   c계열: CPU 집약 (암호화, 연산)
   t계열: 버스터블 (간헐적 부하, 관리용)
3) 서버리스/관리형 우선 — 운영 부담 제거 (Aurora Serverless v2, Fargate, Lambda)


■ On-Prem VPC — EC2 인스턴스 권장 스펙

[MySQL 서버]
  권장: r7g.large (Graviton3, 2 vCPU / 16 GB RAM)
  이유: MySQL InnoDB 버퍼풀은 메모리에 비례해 성능이 올라간다.
        wearable_raw 테이블에 웨어러블 데이터가 상시 스트리밍 write되고
        pii_raw 임시 보관, token_map 조회가 동시에 발생한다.
        메모리 집약 워크로드 → 메모리 최적화 r계열이 맞다.
        r7g는 메모리:vCPU = 8:1 비율로 DB 워크로드 표준이다.
  개발/테스트: t3.medium (2 vCPU / 4 GB)

[토큰화 서비스 — FastAPI, SHA-256 해싱]
  권장: c7g.medium (Graviton3, 1 vCPU / 2 GB RAM)
  이유: SHA-256은 CPU 바운드 연산이다. 요청마다 해시 계산이 발생하고
        Graviton3은 ARM 아키텍처 특성상 암호화 연산 효율이 x86보다 높다.
        메모리는 거의 쓰지 않고 CPU가 병목이므로 c계열이 적합하다.
  개발/테스트: t3.small (2 vCPU / 2 GB, 버스터블)

[Private API — FastAPI + Nginx, MySQL 조회]
  권장: t3.small (2 vCPU / 2 GB RAM, 버스터블)
  이유: 내부 전용 API로 외부 트래픽이 없다.
        MySQL 조회 결과를 반환하는 I/O 바운드 작업이 주이고
        지속적 고부하가 없어 버스터블 인스턴스로 충분하다.
        부하가 생기면 CPU 크레딧으로 순간 처리, 평시에는 저비용 유지.
  부하 증가 시: c7g.medium으로 상향

[Ansible Control Node]
  권장: t3.micro (2 vCPU / 1 GB RAM)
  이유: ansible-playbook을 실행할 때만 CPU를 쓰고 평시에는 거의 유휴 상태다.
        SSM 세션만 유지하면 되므로 최소 스펙으로 비용을 최소화한다.


■ Service DB VPC — 관리형 서비스 스펙

[Aurora PostgreSQL]
  권장: Aurora Serverless v2 (최소 0.5 ACU / 최대 16 ACU)
  이유: 배치 중심 파이프라인(하루 1~2회 ML 학습 결과 적재)이라
        상시 프로비저닝보다 Serverless v2가 비용 효율이 높다.
        2024년 11월부터 scale-to-zero 지원 — 유휴 시 비용 없음.
        부하 시 자동 스케일 업, ML 추론 결과 대량 적재 시 일시적 ACU 증가.
        1 ACU = vCPU + 약 2 GB RAM
        최소 0.5 ACU ≈ 1 GB RAM (대기 상태)
        최대 16 ACU ≈ 32 GB RAM (피크 처리)
  프로비저닝이 필요한 경우: db.r7g.large (Graviton3, 상시 고부하 시)

[RDS Proxy]
  관리형 서비스 — 인스턴스 선택 불필요
  연결 풀링 활성화 권장: Lambda ↔ Aurora 간 커넥션 급증 방지

[DynamoDB]
  서버리스 — 인스턴스 선택 불필요
  On-Demand 모드 권장: 배치성 요청 패턴에 적합, 유휴 비용 없음

[Lambda]
  권장 메모리: 512 MB
  이유: Lambda는 메모리에 비례해 vCPU가 할당된다.
        기본값 128 MB는 Aurora 연결 + JSON 직렬화 시 부족할 수 있다.
        512 MB → 약 0.33 vCPU 할당, 충분한 처리 속도 확보.
        1,792 MB 이상 = 1 vCPU 전체 — 이 워크로드에서는 오버스펙.


■ lifesync360-platform — ECS Fargate 스펙

  현재 설정: 256 CPU (0.25 vCPU) / 512 MB  ← 최소 스펙
  문제점:    FastAPI 기동 시 100~200 MB 기본 사용,
             DB 커넥션 풀 + 동시 요청 처리 시 512 MB 부족 가능성.
             0.25 vCPU는 동시 요청 증가 시 응답 지연 발생.

  권장 (운영): 512 CPU (0.5 vCPU) / 1024 MB
  이유:        FastAPI의 비동기 처리와 동시 요청을 안정적으로 처리하는 최소 운영 스펙.
               메모리 여유로 DB 커넥션 풀 유지 가능.

  부하 증가 시: 1024 CPU (1 vCPU) / 2048 MB
               ECS Auto Scaling과 조합하면 평시 비용 유지하며 피크 대응 가능.


■ 전체 인스턴스 선택 요약

컴포넌트          현재         권장              패밀리 선택 이유
----------------------------------------------------------------------
MySQL EC2         미정         r7g.large         메모리 집약 DB 워크로드
토큰화 EC2        미정         c7g.medium        CPU 집약 암호화 연산

■ Graviton3를 선택하는 이유 (발표용)

"같은 성능을 x86보다 20% 저렴하게 제공합니다.
 AWS 자체 설계 ARM 기반 프로세서로, MySQL, PostgreSQL, FastAPI 등
 이 프로젝트의 모든 주요 워크로드에서 공식 지원됩니다.
 에너지 효율이 60% 높아 지속가능성 측면에서도 유리합니다."


■ Aurora Serverless v2를 선택하는 이유 (발표용)

"이 서비스의 Aurora는 GCP Vertex AI 추론 결과를 받아 저장하는
 배치성 워크로드가 주입니다. 하루 1~2회 대량 적재 외에는 거의 유휴 상태입니다.
 Serverless v2는 유휴 시 scale-to-zero로 비용이 0에 가까워지고
 추론 결과 대량 적재 시 자동으로 스케일 업합니다.
 상시 프로비저닝 인스턴스 대비 이 패턴에서 비용이 60~70% 절감됩니다."

1. ML을 LLM보다 선택한 이유
----------------------------------------------------------------

■ 결론
이 프로젝트의 use case(건강 점수, 금융/보험 추천)는 ML이 LLM보다 명백히 맞는 선택이다.


■ 이유 1 — 데이터 타입 미스매치
LLM은 텍스트/언어 패턴 처리용이다.
이 프로젝트의 입력 데이터는 심박수, SpO2, HRV, 체온(시계열 수치), 나이, 성별, 거래 이력(정형 테이블),
보험 가입 여부, 포인트 잔액(카테고리/숫자)으로 구성된다.
정형 tabular data에는 XGBoost, LightGBM, MLP 같은 ML 모델이 훨씬 정확하고 효율적이다.


■ 이유 2 — 비용 격차
항목          ML                                  LLM
추론 비용     수십만 유저 배치 → 몇 달러          토큰 기반 과금 → 수백~수천 달러
모델 크기     수 MB                               수 GB ~ 수백 GB
GPU 요구      없거나 최소                          필수

배치 잡으로 전체 유저 점수를 계산하는 구조에서 LLM은 비용적으로 비현실적이다.


■ 이유 3 — 설명가능성 (Explainability)
건강 점수(0~100) 산출, 보험 할인율 적용 같은 의사결정에는 근거가 필요하다.
- ML: SHAP value, feature importance로 "심박 변이도가 점수 하락에 가장 영향" 설명 가능
- LLM: 왜 그 점수인지 설명 불가 → 금융/헬스케어 규제 환경에서 치명적


■ 이유 4 — 일관성과 재현성
같은 입력에 대해 항상 같은 출력이 보장되어야 한다.
LLM은 temperature/sampling 때문에 결과가 달라질 수 있으나,
ML 모델은 완전히 결정적(deterministic)이다.


■ 이유 5 — 개인정보 보호
LLM API에 건강 데이터 + 금융 데이터를 전송하면 데이터 컴플라이언스 문제가 발생한다.
Vertex AI Custom Training으로 자체 모델을 학습하면 데이터가 외부 LLM 서비스로 나가지 않는다.


■ 한 줄 요약
건강 점수 계산과 상품 추천은 언어 생성 문제가 아니라 분류/회귀/랭킹 문제다 — ML이 도구로서 맞다.
LLM이 의미 있는 경우는 추천 이유를 자연어로 생성할 때뿐이며,
이마저도 규칙 기반 템플릿으로 충분히 커버 가능하다.

2. VPC 다중 분리 — 설계 이유 및 질의응답
----------------------------------------------------------------

■ VPC 구성 및 역할 (전체 ap-northeast-2 서울 리전)

VPC              역할                         외부 접점 처리 방식
On-Prem VPC      MySQL, Ansible — PII 원본    외부 접점 없음 (완전 격리)
Service DB VPC   Aurora, DynamoDB, Lambda     Lambda만 접근 (프라이빗)
Data VPC         S3, Glue — 정제 파이프라인   Glue Job만 접근 (프라이빗)
Management VPC   운영 채널 (SSM, Bastion)     운영팀만 SSM 접근
Wearable VPC     IoT 웨어러블 데이터 수신     IoT Core + API Gateway로 외부 수신
Group VM VPC     LS 계열사 연동               Site-to-Site VPN으로 계열사 연결

→ 모든 VPC는 서울 리전(ap-northeast-2)에 배치한다.
  "외부"는 다른 리전이 아닌 네트워크 구조(퍼블릭/프라이빗 서브넷, VPN)로 표현한다.


■ Q. VPC를 왜 이렇게 많이 나눴나요? 서브넷으로도 충분하지 않나요?

A. 서브넷 분리는 논리적 분리고, VPC 분리는 물리적 분리입니다.
   서브넷은 같은 라우팅 테이블과 인터넷 게이트웨이를 공유하기 때문에
   보안 그룹 설정 실수 하나로 전체가 뚫릴 수 있습니다.
   VPC는 라우팅 자체가 없으면 물리적으로 통신이 불가능합니다.
   금융/헬스케어 데이터를 다루는 이 서비스는 ISMS-P, 금융보안원 가이드상
   "논리적 분리"로는 감사를 통과하지 못합니다.


■ Q. VPC마다 데이터 민감도가 다르다는 게 무슨 의미인가요?

A. 각 VPC가 보유한 데이터의 민감도와 외부 노출 가능성이 다릅니다.

   On-Prem VPC    → 주민번호, 계좌번호 원본 (PII 최고 등급, 외부 접점 없음)
   Service DB VPC → AES-256 암호화된 PII + 금융 추천 결과 (Lambda만 접근)
   Data VPC       → 비식별 처리된 분석용 데이터 (Glue Job만 접근)
   Wearable VPC   → 외부 IoT 기기에서 직접 데이터 수신 (외부 접점 최대)
   Group VM VPC   → 계열사 외부 시스템 (신뢰도 낮은 외부망)

   민감도가 낮은 VPC(Wearable)가 공격받아도
   TGW 라우팅이 없으면 민감도 높은 VPC(On-Prem, Service DB)로 이동 자체가 불가능합니다.


■ Q. Wearable VPC와 Group VM VPC의 "외부" 접점은 어떻게 구현했나요?

A. 다른 리전이 아닌, 같은 서울 리전 내 네트워크 구조로 외부 경계를 표현했습니다.
   AWS에서 서울 외 다른 리전은 일본, 싱가포르 등 국외 리전이기 때문에
   "외부 시스템 시뮬레이션"을 위해 다른 리전 VPC를 쓰는 패턴은
   공식 AWS 아키텍처 가이드와 실제 사례 어디에도 존재하지 않습니다.

   실제 외부 접점 구현 방식:

   [Wearable VPC — IoT 기기 연결]
   웨어러블 기기 → 인터넷 → AWS IoT Core (퍼블릭 엔드포인트, MQTT)
                                  ↓
                         Wearable VPC (프라이빗 서브넷, 서울)
                                  ↓
                              TGW → Service DB VPC

   [Group VM VPC — 계열사 연결]
   계열사 온프레미스/자체 시스템 → Site-to-Site VPN
                                        ↓
                               서울 TGW
                                        ↓
                              Group VM VPC (서울)


■ Q. VPC를 나누면 비용이 더 들지 않나요?

A. 맞습니다. TGW 비용이 발생합니다.
   동일 리전 내 TGW:  Attachment $0.05/hr + 데이터 $0.02/GB
   6개 VPC 기준 월 attachment 비용 ≈ $216/월

   그러나 이 비용은 보안 사고 발생 시 대응 비용, 규제 위반 과징금,
   서비스 중단으로 인한 손실과 비교하면 보험료 수준입니다.
   실제 금융/헬스케어 서비스에서 VPC 분리는 비용이 아니라 필수 요건입니다.


■ Q. TGW 대신 VPC Peering을 쓰면 더 싸지 않나요?

A. VPC가 3~4개 이하면 Peering이 낫습니다.
   그러나 이 프로젝트처럼 6개 이상이면 Peering은 N*(N-1)/2 개의 연결이 필요합니다.
   6개 VPC 기준으로 15개 Peering 연결을 각각 관리해야 하고,
   라우팅 테이블도 수십 개로 늘어납니다.
   TGW는 허브 앤 스포크 구조로 중앙에서 라우팅을 통제하므로
   VPC 수가 많을수록 TGW가 관리 면에서 유리합니다.


----------------------------------------------------------------
3. "외부 시스템을 다른 리전 VPC로 표현하는 설계"가 잘못된 이유
----------------------------------------------------------------

■ 검증 결과

"외부 시스템 시뮬레이션을 위해 다른 리전에 VPC를 배치하는 패턴"은
AWS 공식 아키텍처 가이드, AWS 사례집, 실제 엔터프라이즈 구현 사례 어디에도
존재하지 않는다.

멀티 리전 VPC가 실제로 사용되는 경우:
  - Active-Active: 글로벌 사용자를 위해 여러 국가에 동일 서비스를 배포할 때
  - Active-Passive: 리전 장애 시 페일오버를 위한 DR 구성
  - 실제로 여러 국가에 사용자와 데이터가 분산된 글로벌 서비스

이 프로젝트는 한국 내 서비스다. 위 케이스에 해당하지 않는다.


■ 왜 이 설계가 논리적으로 맞지 않는가

AWS에서 한국 내 리전은 서울(ap-northeast-2) 하나뿐이다.
"다른 리전"은 곧 "국외"다.

  ap-northeast-1 → 일본 도쿄
  ap-southeast-1 → 싱가포르
  us-east-1      → 미국 버지니아

"외부 접점이니까 다른 리전에 둔다"는 것은
"외부 접점이니까 일본에 둔다"와 같은 말이 된다.

외부(External)의 실제 의미:
  외부 = 인터넷을 통해 들어오는 트래픽 (신뢰도가 낮은 경로)
  외부 ≠ 다른 국가의 AWS 리전

한국에서 웨어러블을 착용한 사용자의 심박 데이터가
일본 리전 VPC를 경유해서 서울로 들어오는 구조는
성능, 비용, 법규 어느 면에서도 정당화되지 않는다.


■ 다른 리전 VPC를 사용했을 때 발생하는 실제 문제

1) 개인정보 국외 이전 위반
   웨어러블 심박수, SpO2, 체온은 민감정보(건강정보)에 해당한다.
   국외 리전 경유 시 개인정보보호법 제28조의8 위반 — 정보주체 동의 또는
   보호위원회 인가 없이는 불가능하다.

2) 비용 이중 과금
   동일 리전:   TGW 데이터 처리 $0.02/GB
   크로스 리전: TGW 처리 $0.02 + 리전 간 전송 $0.02 = $0.04/GB
   상시 스트리밍되는 웨어러블 데이터는 리전 간 이동 시 비용이 두 배가 된다.

3) 레이턴시 증가
   서울 ↔ 도쿄: 약 30~40ms 추가
   실시간 심박 이상 감지 시나리오에서 이 지연은 서비스 품질에 직결된다.

4) 보안 그룹 리전 제한
   보안 그룹은 리전 단위 리소스다.
   다른 리전 VPC의 보안 그룹을 참조할 수 없어
   모든 허용 규칙을 CIDR(IP 대역)로만 관리해야 한다.

5) PrivateLink 크로스 리전 불가
   VPC Endpoint(PrivateLink)는 같은 리전 내에서만 동작한다.
   외부 리전에서 서울 RDS Proxy, API Gateway Private Endpoint에 접근하려면
   퍼블릭 경로를 경유해야 한다.


■ 올바른 설계 — "외부"를 네트워크 구조로 표현

[Wearable VPC: IoT 기기 연결]

  한국 사용자 웨어러블 기기
         ↓ (인터넷, MQTT/HTTPS)
  AWS IoT Core (서울, 퍼블릭 엔드포인트)
         ↓
  Wearable VPC 프라이빗 서브넷 (서울 ap-northeast-2)
         ↓
  TGW → Service DB VPC

  - IoT Core가 외부 기기와의 인증/연결을 담당
  - VPC 내부는 완전 프라이빗, 인터넷 직접 노출 없음
  - 데이터는 서울 리전 내에서만 이동 → 국외 이전 없음

[Group VM VPC: 계열사 연결]

  LS 계열사 온프레미스/자체 시스템
         ↓ (Site-to-Site VPN 또는 Direct Connect)
  서울 TGW
         ↓
  Group VM VPC (서울 ap-northeast-2)

  - VPN/Direct Connect가 "신뢰도 낮은 외부망"과의 경계 역할
  - 계열사 시스템이 실제로도 한국에 있으므로 서울 리전이 맞다
  - 실제 대기업 그룹사 환경의 표준 연결 구조


■ 발표 시 예상 질문 대응

Q. "Wearable VPC와 Group VM VPC를 외부로 취급했다면 다른 리전에 둔 건가요?"
A. "처음 설계 시 '외부 = 다른 리전'으로 잘못 접근했습니다.
   그런데 AWS에서 서울 외 리전은 전부 국외입니다.
   한국 내 서비스에서 웨어러블 건강 데이터가 일본이나 싱가포르 리전을
   경유하면 개인정보 국외 이전 규정 위반이 됩니다.
   실제로 다른 리전 VPC로 외부를 시뮬레이션하는 아키텍처 패턴은
   AWS 공식 가이드와 실사례 어디에도 없습니다.
   올바른 구현은 같은 서울 리전 내에서 IoT Core와 VPN으로
   외부 경계를 처리하는 방식입니다."

Q. "그럼 외부 접점 격리는 어떻게 하나요?"
A. "네트워크 구조로 표현합니다.
   웨어러블 기기는 AWS IoT Core(퍼블릭 엔드포인트)를 통해 인증 후
   프라이빗 서브넷의 Wearable VPC로만 데이터가 전달됩니다.
   IoT Core가 외부와 내부의 경계 역할을 하고,
   VPC 내부는 인터넷에 직접 노출되지 않습니다.
   계열사(Group VM)는 Site-to-Site VPN으로 연결해
   인터넷을 경유하지 않는 전용 암호화 채널을 사용합니다."


----------------------------------------------------------------
4. GCP 리소스를 사용하는 이유 (멀티 클라우드 전략)
----------------------------------------------------------------

■ 전제: 왜 AWS만으로는 부족한가

이 프로젝트는 헬스케어 + 금융이 결합된 서비스다.
핵심 워크로드는 두 가지로 나뉜다:
  A) 민감 데이터 보관 + 트랜잭션 처리 → AWS가 강점
  B) 대규모 분석 + ML 모델 학습/추론  → GCP가 강점

"하나의 클라우드로 모두 해결하는 것"보다
"각 클라우드가 가장 잘 하는 것을 맡기는 것"이 멀티 클라우드의 핵심 논리다.


■ 먼저 확인: Redshift와 SageMaker는 어느 수준까지 처리 가능한가

[Redshift Serverless 2025 기준 실제 스펙]
- 최대 용량:  1024 RPU (2025년 2월 상향, 1 RPU = 2 vCPU + 16GB RAM)
- 최대 스토리지: 8 PB (512 RPU 기준)
- 최소 시작:  4 RPU (2025년 6월 추가, 소규모 워크로드 지원)
- 컬럼 수:   테이블당 최대 1,600개
- 동시 쿼리:  최대 50개 대기열 (멀티 클러스터 포함)
- 과금 단위:  쿼리당 최소 60초 과금

→ "Redshift가 대용량을 못 처리한다"는 말은 틀렸다.
  엔터프라이즈 수준의 대용량 분석도 충분히 처리 가능하다.

[SageMaker 실제 강점]
- Multi-model Endpoint: 하나의 엔드포인트에 여러 모델 동시 호스팅 가능
  (Vertex AI는 이 기능이 기본 미지원 — 커스텀 컨테이너로 직접 구현해야 함)
- 플랫폼 성숙도: 더 긴 운영 이력, 레퍼런스/커뮤니티/공식 문서 풍부
- 커스텀 컨테이너 자유도: 세밀한 인프라 제어 가능
- SageMaker Autopilot: 정형 데이터 AutoML 지원 (Vertex AI AutoML과 동급)

→ SageMaker도 강력한 플랫폼이다. "Vertex AI가 전반적으로 우위"는 과장이다.


■ 그럼에도 BigQuery를 선택한 실제 이유

"Redshift가 못 한다"가 아니라 "이 프로젝트 데이터 파이프라인 구조에서 BigQuery가 더 효율적"이 정확한 표현이다.

1) 크로스 클라우드 전송 이후 파이프라인 단순도 차이
   실제 데이터 흐름:
   AWS On-Prem → S3 Raw → Glue(정제) → S3 Curated → [Storage Transfer] → GCP

   정제는 AWS Glue가 담당하고, 완성된 S3 Curated 데이터를 GCP로 한 번 전송한다.
   크로스 클라우드 전송(S3 → GCS)은 어느 선택을 해도 발생한다.
   차이는 그 이후 구간이다:

   BigQuery + Vertex AI:
     S3 Curated → GCS (1회 전송) → BigQuery → Vertex AI
     GCS 도착 이후로는 GCP 내부에서 추가 복사 없이 ML까지 완결된다.

   Redshift + SageMaker (AWS 단일 클라우드로 구성 시):
     S3 Curated → Redshift (분석용 적재)
     S3 Curated → S3 유지 (SageMaker 학습은 S3에서 직접 읽음)
     데이터가 Redshift와 S3에 이중으로 존재하고, SageMaker 학습 시 S3를 재참조한다.

   파이프라인 단계가 줄수록 장애 지점이 줄고 운영 부담이 낮아진다.

2) 동시 쿼리 처리 방식의 차이
   Redshift Serverless는 동시 대기열이 50개로 제한된다.
   여러 분석 잡이 동시에 실행되는 배치 환경에서 병목이 될 수 있다.
   BigQuery는 슬롯을 자동으로 배분하여 동시 쿼리 수에 실질적 제한이 없다.

3) 쿼리 과금 구조의 차이
   Redshift Serverless는 쿼리당 최소 60초 과금이 적용된다.
   이 프로젝트처럼 짧은 집계 쿼리를 하루 수십 번 실행하는 경우
   실제 처리 시간보다 과금이 더 나올 수 있다.
   BigQuery는 처리한 데이터 용량($5/TB) 기준 과금으로 단기 쿼리에 유리하다.

4) BigQuery ML — 데이터 이동 없이 SQL로 ML 가능
   BigQuery 안에서 SELECT 결과를 바로 모델 학습 데이터로 사용할 수 있다.
   SageMaker는 학습 데이터를 반드시 S3에 별도로 내보내야 하는 단계가 있다.

5) 컬럼 수 한계
   웨어러블 + 금융 + 건강 데이터를 결합한 wide 테이블 구성 시
   Redshift의 1,600개 컬럼 제한이 구조 설계를 제약할 수 있다.
   BigQuery는 테이블당 10,000개 컬럼을 지원한다.


■ Vertex AI를 선택한 실제 이유 (SageMaker와 솔직한 비교 포함)

[Vertex AI가 앞서는 부분]
1) BigQuery 직접 연동
   Vertex AI는 BigQuery 테이블을 학습 데이터 소스로 직접 지정할 수 있다.
   SageMaker는 학습 전 데이터를 S3로 복사하는 단계가 반드시 필요하다.
   → BigQuery에 데이터가 있는 이 프로젝트에서 이 차이가 가장 중요하다.

2) Explainable AI 기본 내장
   Vertex AI는 feature attribution(기여도)을 별도 라이브러리 없이 기본 제공한다.
   SageMaker Clarify로 유사 기능을 제공하지만 별도 설정과 추가 비용이 발생한다.

3) Eventarc 연동으로 이벤트 드리븐 파이프라인 완결
   Vertex AI 추론 완료 → Eventarc 트리거 → Cloud Run → AWS API GW → Lambda → Aurora
   이 흐름이 GCP 내부에서 서버리스로 처리된다.

[SageMaker가 앞서는 부분 — 솔직하게]
- Multi-model Endpoint: Vertex AI가 기본 미지원, SageMaker가 앞선다.
- 플랫폼 성숙도와 레퍼런스: SageMaker가 더 많다.
- AWS 기존 인프라와의 통합: 당연히 SageMaker가 유리하다.

[결론]
Vertex AI가 SageMaker보다 "전반적으로 낫다"는 것이 아니라,
"BigQuery에 데이터가 있는 상황에서 Vertex AI가 파이프라인 상 더 자연스럽다"는 것이다.
SageMaker를 선택했다면 BigQuery 대신 Redshift를 써야 하고,
그러면 GCP 생태계의 통합 이점이 사라진다.


■ GCP vs AWS 주요 리소스 비교 (객관적 평가 포함)

역할            GCP 선택     AWS 대안         GCP 강점                    AWS 강점
-------------------------------------------------------------------------------------
데이터웨어하우스 BigQuery     Redshift         파이프라인 통합, 컬럼 10K   성숙도, 멀티클러스터
ML 학습/추론    Vertex AI    SageMaker        BigQuery 직접 연동          Multi-model EP, 레퍼런스
스트림 처리     Dataflow     Glue/Kinesis     Apache Beam 완전 관리형     Kinesis 실시간 강점
이벤트 트리거   Eventarc     EventBridge      GCP 서비스 네이티브 연동    AWS 서비스 광범위 지원
컨테이너 실행   Cloud Run    Lambda/ECS       VPC Connector 프라이빗 통신 Lambda 생태계 성숙
오브젝트 저장   GCS          S3               GCP 내 전송 무료            리전/스토리지 클래스 다양


■ 멀티 클라우드를 선택한 구조적 이유

업계 현황: 2025년 기준 87~89%의 엔터프라이즈가 2개 이상의 클라우드를 사용한다.
멀티 클라우드는 복잡성을 감수하는 특수한 선택이 아니라 현재 업계 표준이다.

1) 규정 준수와 데이터 주권 [가장 중요]
   한국 금융/개인정보 관련 규정(개인정보보호법, 신용정보법)은
   원본 PII 및 금융 거래 데이터를 국내 서버에 보관하도록 요구한다.
   → 원본 데이터: AWS 서울 리전(ap-northeast-2) 보관 — 규정 준수
   → 비식별 분석 데이터만 GCP로 이전 — 규정 리스크 없음
   AWS만 쓰면 이 요건을 만족하지만, GCP만 쓰면 원본 데이터를 한국 외부로 내보낼 수 없다.

2) 기존 인프라 활용
   계열사(은행, 카드, 보험)가 이미 AWS 기반 시스템을 운영하고 있다.
   Cognito 인증, ls_user_id 체계, 기존 RDB를 그대로 사용하면서
   분석/ML만 GCP로 확장하는 것이 전면 마이그레이션보다 현실적이고 비용도 낮다.

3) 워크로드 특성에 맞는 과금 모델 분리
   항상 켜져 있어야 하는 트랜잭션 DB, API 서버 → AWS (예약 인스턴스 할인 활용)
   필요할 때만 실행하는 ML 학습, 대용량 배치 분석  → GCP 온디맨드 (유휴 비용 없음)
   GCP ML 마이그레이션으로 컴퓨팅 비용 25% 절감한 사례가 실제로 존재한다.

4) 벤더 락인 방지
   분석/ML 파이프라인이 GCP에 있으면 AWS가 요금 인상이나 서비스 변경을 해도
   핵심 분석 워크로드는 독립적으로 유지된다.

5) 역할 분담이 명확하다
   AWS: 사용자 인증, PII 보관, 트랜잭션 처리, 서비스 API 제공
   GCP: 데이터 정제, 대규모 분석, ML 학습/추론, 결과 반환
   두 클라우드의 책임 영역이 명확히 나뉘어 운영 주체도 분리 가능하다.


■ 멀티 클라우드의 단점 (솔직하게)
- 운영 복잡도 상승: AWS + GCP 양쪽을 숙지해야 함
- 네트워크 비용: Cloud VPN 또는 Dedicated Interconnect 구간 비용 발생
- IAM 정책 이원화: 두 클라우드 권한 관리를 각각 해야 함
- 장애 추적 어려움: 문제 발생 시 두 클라우드를 동시에 들여다봐야 함

→ 이 단점들은 실재한다. 그러나 이 프로젝트에서 멀티 클라우드를 선택한 이유는
  "GCP가 AWS보다 낫기 때문"이 아니라
  "데이터 규정상 AWS는 유지해야 하고, 분석/ML 파이프라인은 GCP 생태계에서
  더 단순하고 효율적으로 구성되기 때문"이다.


■ 핵심 요약
Redshift와 SageMaker도 충분히 강력하다. AWS만으로도 기술적으로는 구현 가능하다.
그러나 이 프로젝트는 다음 두 가지 제약 조건 때문에 멀티 클라우드를 선택했다:

  첫째, 한국 금융/개인정보 규정으로 원본 데이터는 AWS 서울 리전에 있어야 한다.
  둘째, 분석/ML 파이프라인은 GCS → Dataflow → BigQuery → Vertex AI로 구성할 때
        데이터 이동 없이 GCP 내부에서 완결되어 운영 복잡도와 비용이 낮다.

"GCP가 더 좋아서"가 아니라 "규정 준수 + 파이프라인 효율 + 기존 인프라 활용"의
세 가지 조건을 동시에 만족하는 선택이 멀티 클라우드다.


----------------------------------------------------------------
5. Vertex AI vs Gemini — "Gemini로 바꾸면 안 되나?" 질문에 대한 답
----------------------------------------------------------------

■ 먼저 알아야 할 현재 상황 (2026년 4월 기준)

Google Cloud Next 2026에서 Vertex AI가 "Gemini Enterprise Agent Platform"으로 리브랜딩됐다.

  Gemini Enterprise Agent Platform
    ├── 기존 Vertex AI ML 기능  ← 그대로 유지, 변경 없음
    ├── Agentspace              ← 새로 통합
    └── Gemini Code Assist      ← 새로 통합

"Vertex AI가 없어지고 Gemini가 대체한다"는 오해가 생기기 쉬운데,
실제로는 Vertex AI 기능이 그대로 유지되면서 Gemini 브랜드 아래 확장된 것이다.
기존 Vertex AI 워크로드는 코드 변경 없이 동일하게 동작한다.


■ 두 도구가 푸는 문제 자체가 다르다

Vertex AI AutoML
  목적:  데이터를 학습해서 숫자/분류값을 예측하는 모델을 만드는 것
  입력:  BigQuery 테이블, CSV (정형 데이터)
  출력:  점수(숫자), 등급(분류), 순위(랭킹)
  방식:  수만~수백만 건의 과거 데이터에서 패턴을 학습 → 새 데이터 예측

Gemini API
  목적:  프롬프트에 대한 자연어 텍스트를 생성하는 것
  입력:  텍스트, 이미지, 코드 (비정형)
  출력:  생성된 텍스트, JSON 형태의 답변
  방식:  수천억 개 파라미터로 사전 학습된 언어 모델이 확률적으로 다음 토큰을 생성


■ 건강 점수 계산에 Gemini를 쓰면 생기는 문제

[Vertex AI AutoML 방식]
  입력: heart_rate=72, spo2=97, hrv_ms=45, steps=8200
  출력: health_score=78, grade="GOLD"

  - 10만 명 실데이터를 학습해서 수치 패턴을 내재화한 모델이 점수를 계산
  - 같은 입력 → 항상 같은 점수 (완전히 결정적, 재현 가능)
  - 왜 78점인지 feature importance로 설명 가능 (심박 변이도 기여도 32% 등)
  - 100만 건 배치 처리 비용: 수 달러

[Gemini API 방식]
  입력: "심박수 72, SpO2 97, HRV 45ms, 걸음수 8200인 사람의 건강 점수를 알려줘"
  출력: "이 사용자의 건강 점수는 약 75~80점 수준으로 추정됩니다..."

  문제 1: 실행마다 다른 점수가 나올 수 있다 (확률적 생성 구조)
  문제 2: 왜 그 점수인지 근거를 제시하지 못한다 (hallucination 리스크)
  문제 3: Google 공식 문서에 명시: "구조화 출력은 의미적 정확성을 보장하지 않는다"
  문제 4: 100만 건 배치 처리 비용: 토큰 과금으로 수백~수천 달러
  문제 5: 금융/헬스케어 감사 시 "AI가 그렇게 판단했습니다" — 설명 불가


■ 두 도구의 강점 비교

기준                 Vertex AI AutoML          Gemini API
------------------------------------------------------------------
정형 데이터 예측     압도적으로 적합            구조적으로 부적합
재현성               완전 결정적 (항상 동일)    확률적 (매번 달라짐)
설명가능성           Feature attribution 내장   불가
대용량 배치 비용     저렴                       고가 (토큰 과금)
BigQuery 직접 연동   지원                       미지원
자연어 생성          불가                       압도적으로 적합
문서 요약/분석       불가                       적합
빠른 프로토타이핑    학습 시간 필요             빠름


■ 이 프로젝트에서 각각의 역할

Vertex AI AutoML (Gemini Enterprise Platform 내)  ← 핵심 사용처
  - 건강 점수 산출: 웨어러블 수치 → 0~100 점수 (회귀)
  - 등급 분류: 건강 점수 → PLATINUM/GOLD/SILVER/BRONZE/BASIC (분류)
  - 상품 추천 랭킹: 고객 프로필 → 추천 상품 순위 (랭킹)

Gemini API  ← 선택적, 부가적 사용처 (없어도 무방)
  - 추천 이유 자연어 생성
    예: "최근 HRV 수치가 15% 하락하여 심혈관 주의가 필요합니다"
  - 단, 이것도 규칙 기반 템플릿으로 대체 가능하므로 필수는 아님


■ 발표 시 예상 질문 대응

Q. "Vertex AI가 오래됐는데 최신 Gemini를 쓰는 게 낫지 않나요?"
A. "Vertex AI는 2026년 4월 Gemini Enterprise Agent Platform으로 통합됐고
   기능은 그대로 유지되고 있습니다. Gemini API는 텍스트 생성 도구이고
   Vertex AI AutoML은 정형 데이터에서 숫자를 예측하는 도구입니다.
   건강 점수 계산은 언어 생성 문제가 아니라 수치 예측 문제이기 때문에
   Vertex AI AutoML이 맞는 선택입니다."

Q. "Gemini도 숫자 출력이 되지 않나요?"
A. "출력 형식은 가능하지만 근거가 없습니다. Gemini는 학습 데이터에서
   패턴을 학습한 게 아니라 언어 패턴으로 그럴듯한 숫자를 생성합니다.
   금융/헬스케어 서비스에서는 같은 입력에 같은 결과가 나와야 하고,
   왜 그 점수인지 설명할 수 있어야 규제 요건을 충족합니다.
   Gemini는 이 두 가지를 보장하지 않습니다."


■ 한 줄 요약
Vertex AI = 데이터로 패턴을 학습해서 숫자를 예측하는 도구
Gemini    = 언어 패턴으로 텍스트를 생성하는 도구
건강 점수와 상품 추천 점수를 "예측"해야 하는 이 프로젝트에는 Vertex AI가 맞다.
